# MergeCraft Model Card

## Model Summary

| Field | Value |
|---|---|
| **Model name** | MergeCraft-7B |
| **Base model** | Qwen2.5-7B-Coder-Instruct |
| **Fine-tuning method** | LoRA (rank 64) + DeepSpeed ZeRO-3 |
| **Training stages** | SFT → GRPO → DPO (3-stage) |
| **Primary task** | Open source contribution generation with merge-rate optimization |
| **Reward signal** | Simulated maintainer merge probability (trained on 500k labeled PR outcomes) |
| **Training hardware** | 18× NVIDIA A6000 (48GB each), 864GB total VRAM |
| **Training data** | 500k+ labeled PR outcomes + CONTRIBUTING.md corpus |
| **Developed by** | Caleb Newton (USC) |
| **License** | Apache 2.0 |
| **HuggingFace** | `calebnewtonusc/mergecraft-7b` |

## Intended Use

### Primary Use Cases
- Generating open source PRs for specific repositories
- Learning project conventions before contributing
- Right-sizing contributions (splitting large changes into appropriate PRs)
- Writing compelling PR descriptions that get maintainer attention
- Checking contributions against project rules before submitting

### Target Users
- Software engineers making first contributions to unfamiliar projects
- Developer relations teams contributing to OSS on behalf of organizations
- Automated contribution pipelines (OSS bots with human oversight)
- Researchers studying OSS contribution patterns

### Out-of-Scope Uses
- Generating spam contributions to game metrics
- Contributions to private repositories without authorization
- Generating contributions that violate project licenses or codes of conduct

## Training Details

### Stage 1: SFT
- **Data**: 500k labeled (task, merged_PR) pairs
- **Epochs**: 3
- **Duration**: ~6 hours on 18× A6000

### Stage 2: GRPO
- **Reward**: Maintainer simulator merge probability
- **Group size G**: 8 candidates per task
- **Duration**: ~4 hours on 18× A6000

### Stage 3: DPO
- **Preference**: correct + clean > correct + messy
- **Beta**: 0.1
- **Duration**: ~2 hours on 18× A6000

## Known Limitations

### Repository-Specific Blind Spots
- New repos with <50 PRs: insufficient convention data; model falls back to general heuristics
- Repos with unusual conventions not well-represented in training data
- Non-English CONTRIBUTING.md files have lower extraction quality

### Language Limitations
- Python and JavaScript are best-covered (60% of training data)
- Rust contributions perform reasonably well (~80% of Python quality)
- Less common languages (Haskell, Erlang, OCaml) are weaker

### Code Generation Limits
- MergeCraft generates correct contributions for small-to-medium tasks (<500 LOC)
- Large refactors (>1000 LOC) may have correctness issues — use with human review
- Does not replace code correctness testing; always run the test suite

### Maintainer Simulator Limitations
- The simulator is trained on historical data — cannot predict policy changes in project governance
- Projects that have recently changed maintainers may have outdated convention models
- Highly opinionated maintainers with unusual preferences may not be captured

## Ethical Considerations

### Contribution Volume and Maintainer Burden
Mass-generating PRs can burden maintainers. MergeCraft is designed for quality over quantity. Users should:
- Limit to 1-3 PRs per day per repository
- Always read the CONTRIBUTING.md and issue tracker before contributing
- Use MergeCraft as a quality enhancer, not a spam generator

### Attribution
All contributions generated by MergeCraft should be reviewed, understood, and owned by the human submitter. Do not submit AI-generated contributions without review.

### Compliance
Always ensure contributions comply with project licenses. MergeCraft does not automatically check license compatibility between new code and project licenses.
